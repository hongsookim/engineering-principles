# BOOK BIBLE

## Engineering Principles — 기술 리더의 의사결정 원칙

**문서 버전**: 1.0
**최종 수정**: 2026-04-06
**작성자**: Creative Director
**용도**: 모든 에이전트의 Single Source of Truth. 집필, 편집, 검수의 모든 단계에서 이 문서를 기준으로 판단한다.

---

## 1. 비전 선언문

### 이 책이 존재하는 이유

기술은 변한다. 도구는 변한다. 하지만 좋은 판단의 기준은 변하지 않는다.

이 책은 기술 리더가 매일 마주하는 수백 개의 판단 — 기술 선택, 인력 배치, 속도와 품질의 균형, 조직 구조 설계 — 에 대한 **판단의 운영 체제**를 제공한다. 레이 달리오가 『Principles』에서 삶과 일의 원칙을 체계화했듯이, 이 책은 엔지니어링 리더십의 원칙을 체계화한다.

### 핵심 가치 선언

> **원칙은 기술보다 오래 살아남는다.**

1. 기술 리더의 의사결정에서 원칙이 중요하다. 원칙 없는 판단은 감각에 의존하고, 감각은 체계적으로 틀린다.
2. AI 시대에도 원칙의 중요성은 변하지 않는다. 도구가 강력해질수록, 도구를 다루는 사람의 판단력이 더 중요해진다.
3. 원칙은 제약이 아니라 운영 체제다. 좋은 운영 체제 위에서는 어떤 기술 변화에도 올바른 판단을 내릴 수 있다.

### 이 책이 전달하는 하나의 문장

"불완전한 원칙이라도, 원칙이 없는 것보다 언제나 낫다."

## 2. 대상 독자

### 주 독자 프로필

| 구분 | 설명 |
|------|------|
| **직함** | CTO, VP of Engineering, Engineering Manager, Tech Lead, CDAO, Head of Data/AI |
| **경력** | 7년 이상의 엔지니어링 경력. 최소 1개 팀 이상을 이끌고 있거나, 이끌 준비를 하는 사람 |
| **환경** | 빠르게 성장하는 기술 조직. 스타트업~미드사이즈 기업. 기술 변화가 빈번한 환경 |
| **언어** | 한국어 사용자. 영어 기술 용어에 익숙 |

### 그들의 고민

1. **판단의 일관성 부재**: 매번 백지에서 사고를 시작한다. 비슷한 상황에서 다른 판단을 내린 경험이 있다.
2. **기술 변화에 대한 불안**: AI를 비롯한 새로운 기술이 등장할 때마다 "도입해야 하나, 관망해야 하나"로 흔들린다.
3. **실행자에서 판단자로의 전환 고통**: 코드를 짜는 것에서 판단을 내리는 것으로 역할이 바뀌었는데, 판단의 프레임워크가 없다.
4. **조직 확장의 병목**: 리더 본인이 모든 판단에 관여하지 않으면 조직이 느려진다. 원칙이 내재화되지 않았기 때문이다.
5. **속도와 품질 사이의 긴장**: "빠르게 움직이되 깨뜨리지 말라"를 구호가 아닌 구조로 만들고 싶다.

### 이 책이 해결하는 문제

- 판단의 프레임워크를 제공하여, 매번 백지에서 시작하지 않게 한다.
- 기술 변화 앞에서 흔들리지 않는 기준을 제시한다.
- 원칙을 조직에 내재화하는 구조적 방법론을 안내한다.
- 저자의 실전 경험(토스 CDAO로서의 Data First → AI Native 전환)을 통해 원칙의 현실적 적용을 보여준다.

### 부 독자

- **시니어 엔지니어**: 리더로의 전환을 준비하는 사람. 판단의 기준을 미리 세우고 싶은 사람.
- **프로덕트 매니저/비즈니스 리더**: 기술 리더의 사고방식을 이해하고 싶은 사람.

## 3. 톤 & 스타일 가이드

### 참조 모델

레이 달리오 『Principles: Life and Work』의 문체를 차용하되, 엔지니어링 맥락에 맞게 조정한다.

### 시점

**1인칭 단수 (나)**. 저자 김홍수의 경험과 관점에서 서술한다. "나는 ~했다", "나의 판단은 ~이었다", "나는 이렇게 생각한다."

### 어조 규칙

| 원칙 | 예시 |
|------|------|
| **직설적** | "이것은 효과적이지 않다" (O) / "이것은 약간의 개선 여지가 있을 수 있다" (X) |
| **선언적** | "원칙은 기술보다 오래 살아남는다." 결론을 먼저, 근거를 뒤에. |
| **명령형** | "현실을 직시하라." "문제를 먼저 정의하라." 원칙의 제목과 하위 항목은 명령형으로 작성. |
| **구체적** | 추상적 교훈 대신 구체적 상황과 판단을 제시. |
| **간결** | 한 문장에 하나의 생각. 불필요한 수식어 제거. |

### 문체 세부 규칙

1. **문장 길이**: 평균 20~40자. 긴 문장은 끊는다. 짧은 문장과 긴 문장을 교차하여 리듬감을 만든다.
2. **단락 구성**: 한 단락에 하나의 생각. 단락은 3~5문장 이내.
3. **접속사 사용**: "하지만", "그리고", "따라서"로 시작하는 문장 허용. 달리오 스타일의 논리적 흐름 유지.
4. **강조 표기**: 핵심 개념은 **굵게**. 최소한으로 사용하되, 핵심 선언에는 반드시 적용.
5. **인용**: 달리오, 베조스 등 참고 인물의 발언은 이탤릭(*기울임*)으로 처리하고, 저자의 재해석을 바로 이어 붙인다.
6. **블록 인용(>)**: 저자의 핵심 선언문, 관통 원칙에 사용. 남발 금지.
7. **영문 병기**: 핵심 기술 용어는 첫 등장 시 한글(영문) 형식으로 병기. 이후 한글 또는 영문 중 자연스러운 쪽을 택한다. 예: "기술 부채(Technical Debt)", "장애 회고(Post-mortem)"
8. **존칭**: 독자에게 존칭 사용하지 않는다. 평서문 + 명령형. "~하라", "~마라"로 통일.

### 핵심 구문 패턴

달리오 스타일의 반복 가능한 구문 패턴:

```
[선언형 원칙]
[왜 그런지 1~2문장 설명]
[구체적 하위 항목 a), b), c)...]
```

- **대조 구문**: "A는 ~이다. 하지만 B는 ~이다."
- **정의 구문**: "~한다는 것은 ~이 아니라, ~이다."
- **반복 강조**: "이것은 A에도, B에도, C에도 동일하게 적용된다."
- **자문 구문**: "~인가? 그렇다면 ~이다."
- **3요소 나열**: "A, B, C — 이것들은 ~이다." (3개 항목 병렬 후 총괄)

### 하지 말 것 리스트

| 금지 사항 | 이유 |
|-----------|------|
| 학술적 문체 금지 | "~라고 사료된다", "~라 할 수 있겠다" 등 논문체 사용 금지. 현장의 언어로 쓴다. |
| 클리셰 비유 금지 | "숲을 보지 못하고 나무만 본다" 같은 상투적 표현 사용 금지. 비유를 쓸 경우 엔지니어링 맥락에서 꺼낸다. |
| 수동태 최소화 | "결정이 내려졌다" (X) → "나는 결정을 내렸다" (O). 주어를 명확히 한다. |
| 과도한 겸손 금지 | "제가 감히 말씀드리자면" 같은 표현 금지. 저자는 자신의 경험에 기반하여 확신 있게 서술한다. |
| 유행어 남발 금지 | "디지털 트랜스포메이션", "패러다임 시프트" 등 실체 없는 유행어 최소화. 쓸 경우 구체적 의미를 부여한다. |
| 나열식 설명 금지 | "첫째~, 둘째~, 셋째~" 식의 기계적 나열 금지. 각 항목이 논리적 흐름으로 연결되어야 한다. |
| 감탄사/감정 표현 금지 | "놀랍게도!", "안타깝게도..." 등의 감정적 수식어 금지. 사실과 판단으로 서술한다. |
| 독자 아첨 금지 | "여러분도 잘 아시겠지만" 같은 표현 금지. 독자를 동등한 기술 리더로 대한다. |

## 4. 서사 구조

### 전체 구조: 3+1 파트

```
Part I.  내가 어디에서 왔는가        ← 배경: 왜 이 원칙들이 나왔는가
Part II. 엔지니어링 원칙 (5개)       ← 개인의 판단 기준
Part III. 조직 원칙 (8개)            ← 조직의 판단 기준
Part IV. 원칙을 적용하는 법          ← 실전 적용
성찰                                ← 마무리
부록                                ← 원칙 요약표
```

### 각 파트의 역할과 독자 여정

#### Part I. 내가 어디에서 왔는가 (1~2페이지)

**역할**: 저자의 신뢰성 확보 + 원칙의 기원 제시
**독자 여정**: "이 저자는 나와 비슷한 고민을 했고, 그 고민에서 원칙이 탄생했다"
**핵심 서사**: 실행자 → 판단자로의 전환. 원칙 없이 판단하던 시절의 실패. 원칙이 필요하다는 깨달음.
**분량**: 짧고 강렬하게. 3개 섹션 이내. 자기소개가 아니라 원칙의 기원 이야기.

- "실행자에서 판단자로" — 역할 전환의 경험
- "원칙이 필요한 순간" — 하루의 판단들이 축적되어 조직의 방향이 된다
- "왜 원칙인가" — 달리오 인용 + 저자의 재정의

#### Part II. 엔지니어링 원칙 (가장 두꺼운 파트)

**역할**: 개인 차원의 판단 기준 제시. 기술 리더 한 사람이 따르는 원칙.
**독자 여정**: "판단을 내릴 때 이 프레임워크를 적용할 수 있다"
**구성**: 5개 주 원칙, 각 원칙 당 3~5개 하위 원칙
**핵심 아크**: 현실 인식(1) → 문제 해결 프로세스(2) → 본질 구별(3) → 책임의 원칙(4) → 판단의 체계화(5)

5개 원칙의 내적 논리:
1. **현실을 직시하라** — 판단의 출발점. 현실을 왜곡하면 모든 판단이 틀어진다.
2. **5단계 프로세스로 해결하라** — 현실을 직시한 후, 문제를 체계적으로 풀어가는 방법.
3. **변하는 것과 변하지 않는 것을 구별하라** — 문제를 풀 때 무엇에 집중할지의 기준.
4. **이해하지 못하는 것은 배포할 수 없다** — 모든 원칙을 관통하는 책임의 선언.
5. **판단을 체계화하라** — 위 원칙들을 반복 가능하게 만드는 메타 원칙.

#### Part III. 조직 원칙 (두 번째로 두꺼운 파트)

**역할**: 조직 차원의 판단 기준 제시. 개인의 원칙을 조직 수준으로 확장.
**독자 여정**: "좋은 원칙도 조직 구조가 뒷받침하지 않으면 작동하지 않는다"
**구성**: 3개 테마 × 8개 주 원칙

| 테마 | 원칙 | 핵심 질문 |
|------|------|-----------|
| **문화를 올바르게 만들기 위하여** | 6. 급진적 투명성 | 정보가 왜곡 없이 흐르는가? |
| | 7. 실수로부터 배우는 문화 | 학습이 구조화되어 있는가? |
| | 8. 자율과 책임의 균형 | 규칙이 시스템으로 강제되는가? |
| **사람을 올바르게 세우기 위하여** | 9. WHO > WHAT | 올바른 사람이 올바른 자리에 있는가? |
| | 10. 훈련, 평가, 성장 | 사람에 대한 투자가 체계적인가? |
| **시스템을 구축하고 진화시키기 위하여** | 11. 조직을 기계로 바라보라 | 조직의 병목을 진단하고 있는가? |
| | 12. 모든 사람을 빌더로 | 생산성이 민주화되어 있는가? |
| | 13. 도구와 프로토콜 | 원칙이 시스템으로 작동하는가? |

**핵심 아크**: 문화(토양) → 사람(씨앗) → 시스템(구조). 이 순서가 중요하다. 문화 없이 사람을 세울 수 없고, 사람 없이 시스템을 구축할 수 없다.

#### Part IV. 원칙을 적용하는 법

**역할**: 원칙의 실전 사용법. 원칙 간 충돌 해결, 원칙의 진화, 원칙의 한계.
**독자 여정**: "원칙을 알았다. 이제 어떻게 쓰는가?"
**구성**:
- 원칙은 알고리즘이다 — 반복 가능한 판단 체계
- 원칙 간의 충돌을 다루는 법 — 원칙 4가 최상위
- 원칙을 진화시키는 법 — 고정 교리가 아닌 살아 있는 체계

#### 성찰 (마무리)

**역할**: 책 전체의 메시지를 한 문단으로 압축.
**핵심 문장**: "원칙은 기술보다 오래 살아남는다."
**분량**: 1~2페이지. 여운을 남기되 늘어지지 않는다.

#### 부록

**역할**: 빠른 참조용 원칙 요약표.
**구성**: 원칙 번호 | 원칙명 | 핵심 질문 — 표 형식.

### 파트 간 연결고리

| 전환 | 연결 논리 |
|------|-----------|
| Part I → Part II | "경험에서 원칙이 나왔다. 그 원칙을 공유한다." |
| Part II → Part III | "개인의 원칙이 아무리 훌륭해도, 조직의 구조가 뒷받침하지 않으면 실현되지 않는다." |
| Part III → Part IV | "원칙을 알았다면, 이제 적용해야 한다. 적용에는 기술이 필요하다." |
| Part IV → 성찰 | "원칙은 완성형이 아니다. 현실에서 진화한다. 그것이 원칙의 힘이다." |


### 관통 원칙의 역할

**원칙 4: "만든 사람이 이해하지 못하는 것은 배포할 수 없다"**

이 원칙은 단독으로 존재하는 것이 아니라, 모든 원칙의 최종 검증 기준으로 기능한다.

- Part II에서: 개인의 판단에 대한 책임 → "이해하지 못하는 판단을 내리지 마라"
- Part III에서: 조직의 구조에 대한 책임 → "이해하지 못하는 구조를 도입하지 마라"
- Part IV에서: 원칙 간 충돌 시 최우선 기준 → "다른 원칙과 충돌할 때, 원칙 4가 우선한다"

이 원칙이 AI 시대에 더욱 긴급해진 이유:
- 생성의 장벽이 낮아졌다 → 이해 없이 만들기가 쉬워졌다
- 결과물에 대한 책임은 도구가 아니라 사람에게 있다
- 속도를 위해 이해를 포기하면, 더 이상 엔지니어가 아니라 오퍼레이터가 된다

### 주제 흐름 (독자 여정 요약)

```
[나는 왜 원칙이 필요했는가]
         ↓
[개인의 판단 기준을 세운다]
    현실 → 프로세스 → 구별 → 책임 → 체계화
         ↓
[조직의 판단 기준을 세운다]
    문화 → 사람 → 시스템
         ↓
[원칙을 실전에서 쓴다]
    적용 → 충돌 해결 → 진화
         ↓
[원칙은 기술보다 오래 살아남는다]
```

## 6. 원칙 번호 체계 표준

### 달리오 스타일 번호 매기기

레이 달리오의 『Principles』에서 사용한 계층적 번호 체계를 차용한다.

#### 3단계 계층 구조

| 계층 | 형식 | 예시 | 용도 |
|------|------|------|------|
| **주 원칙** | 아라비아 숫자 | 1, 2, 3... 13 | 챕터 수준의 대원칙 |
| **하위 원칙** | 주 원칙.소수점 | 1.1, 1.2, 4.3 | 대원칙 아래의 구체적 원칙 |
| **세부 항목** | 알파벳 소문자 + 괄호 | a), b), c)... | 하위 원칙의 실행 지침/구체적 행동 |

#### 번호 매기기 규칙

1. **주 원칙 번호는 1~13까지 통번호**를 사용한다. Part II(1~5)와 Part III(6~13)에 걸쳐 연속된다.
2. **하위 원칙 번호는 주 원칙 번호를 접두어로** 사용한다. 예: 원칙 4의 세 번째 하위 원칙 = 4.3
3. **세부 항목은 하위 원칙 내에서 a)부터 시작**한다. 각 하위 원칙마다 a)로 리셋된다.
4. **Part III의 테마 제목**("문화를 올바르게 만들기 위하여" 등)은 번호를 매기지 않는다. 테마는 원칙들의 논리적 그룹핑이지, 별도의 원칙이 아니다.

#### 형식 예시

```markdown
## 4. 만든 사람이 이해하지 못하는 것은 배포할 수 없다

[원칙 4의 도입부 — 1~3문단]

### 4.1 이해 없는 속도는 속도가 아니다

[하위 원칙 설명 — 1~2문단]

a) 생성의 장벽이 낮아질수록, 검증의 기준은 높아져야 한다.

b) 검증 없는 생산은 쓰레기를 더 빨리 만드는 것일 수 있다.

c) 속도를 위해 이해를 포기하는 순간, 당신은 더 이상 엔지니어가 아니라 오퍼레이터가 된다.

### 4.2 결과물에 대한 책임은 도구에 전가할 수 없다
...
```

#### 참조 방식

본문에서 다른 원칙을 참조할 때:
- "원칙 4에 따르면" — 주 원칙 참조
- "2.3(근본 원인 진단)에서 다루었듯이" — 하위 원칙 참조. 괄호 안에 제목 약칭 병기.
- 세부 항목(a, b, c)은 본문에서 직접 참조하지 않는다. 참조가 필요하면 하위 원칙 수준에서 참조한다.

## 7. 참고 소스 활용 가이드

### 토스 아티클 활용 원칙

**출처**: 토스 기술 블로그 "Data First, 그리고 AI Native" (https://toss.im/career/article/data_first_ai_native)

이 아티클은 저자의 실전 경험을 뒷받침하는 핵심 소스다. 직접 인용이 아니라, 원칙의 현실적 근거로 자연스럽게 녹여야 한다.

#### 활용 매핑

| 아티클 내용 | 대응 원칙 | 활용 방법 |
|------------|----------|-----------|
| Data First → AI Native 조직 진화 | Part I (배경), 원칙 3 (변하는 것/변하지 않는 것), 원칙 11 (조직 진화) | 저자가 실제로 경험한 조직 전환의 사례로 활용. "나는 토스에서 이 전환을 직접 이끌었다"는 1인칭 서사. |
| "만든 사람이 이해하지 못하는 코드는 배포할 수 없다" | **원칙 4** (관통 원칙) | 이 문장은 책 전체의 관통 원칙. 아티클에서의 발언을 원칙의 기원으로 서사화. |
| 표준화 → 메타화 → 추상화 | 원칙 11 (단계적 진화), 원칙 13 (도구/프로토콜) | "단계를 건너뛸 수 없다"(11.2)의 구체적 사례. 데이터 조직 진화의 3단계로 제시. |
| Agent Builder 플랫폼 | 원칙 12 (모든 사람을 빌더로) | "누구나 Agent를 만들 수 있는 플랫폼"이 12.1(생산성 민주화)의 실현 사례. |
| 설계와 검증 능력 > 직무 경계 | 원칙 9 (WHO > WHAT), 원칙 10 (평가 기준 재정의) | 채용과 평가의 기준이 바뀌어야 한다는 주장의 근거. 9.2의 핵심 논거. |
| 자율과 책임의 균형 + Guardrail | 원칙 8 (자율과 책임), 원칙 4 | 토스의 거버넌스 시스템을 8.2(속도와 안전)의 실례로 활용. |
| 직군별 표준화된 가이드 | 원칙 8 (규칙 명문화), 원칙 13 (프로토콜) | "마크다운 기반 규칙 문서화"를 13.2(프로토콜)의 구체적 실행 사례로 활용. |

#### 활용 시 주의사항

1. **아티클 직접 인용 최소화**. 아티클의 문장을 그대로 가져오지 않는다. 저자의 1인칭 경험으로 재구성한다.
2. **회사명은 절제하여 사용**. "토스에서"라는 표현을 매 문단마다 반복하지 않는다. Part I에서 배경을 충분히 설정한 후, Part II~III에서는 "나의 조직에서", "우리 팀에서" 등으로 자연스럽게 녹인다.
3. **홍보가 아니라 원칙의 근거**로 사용한다. 아티클 내용은 "토스가 이렇게 잘한다"가 아니라 "이 원칙이 현실에서 이렇게 작동했다"의 근거다.

### 저자 소개 가이드

저자 소개는 **3줄 이내**로 간결하게.

```
김홍수. 토스(Toss) CDAO(Chief Data & AI Officer).
데이터 조직의 실행자로 출발하여, 설계자와 전략가로 역할을 확장해왔다.
데이터 전략, AI 설계, 기술-프로덕트-비즈니스의 연결이라는 세 축에서
기술 리더십을 실천하고 있다.
```

- 경력 나열 금지. 현재 역할과 핵심 역량만 기술.
- Part I "내가 어디에서 왔는가"가 저자 소개의 확장판 역할을 하므로, 별도의 긴 소개는 불필요.

### 달리오 『Principles』 참조 가이드

- 달리오의 문체, 구조, 번호 체계를 **차용**하되, 내용은 엔지니어링 맥락으로 완전히 재구성.
- 달리오의 원칙을 직접 인용할 때는 이탤릭으로 표기하고, 바로 이어서 저자의 재해석을 제시.
- "달리오는 ~라고 했다. 나는 엔지니어링의 맥락에서 이렇게 바꾸고 싶다." — 이 구문 패턴 활용.
- 달리오 원칙과의 대응 관계를 독자가 알 수 있도록, 필요한 곳에서 간결하게 언급. 단, 대조 분석으로 흐르지 않는다.

## 8. 품질 기준

### 각 원칙이 반드시 갖춰야 할 요소

모든 주 원칙(1~13)은 다음 5개 요소를 반드시 포함해야 한다.

| 요소 | 설명 | 검증 질문 |
|------|------|----------|
| **1. 선언** | 원칙의 핵심을 한 문장으로 선언 | 원칙 제목만 읽어도 무엇을 말하는지 알 수 있는가? |
| **2. 근거** | 왜 이 원칙이 중요한지 1~3문단으로 설명 | "왜?"에 답하고 있는가? 독자가 납득할 수 있는가? |
| **3. 하위 원칙** | 3~5개의 하위 원칙으로 구체화 | 추상적 선언이 실행 가능한 수준으로 분해되었는가? |
| **4. 세부 항목** | 각 하위 원칙의 구체적 행동 지침 a), b), c)... | 독자가 내일 당장 적용할 수 있는 행동이 있는가? |
| **5. 현실 연결** | 저자의 경험 또는 구체적 상황과의 연결 | 공허한 교훈이 아니라 현장에서 검증된 원칙인가? |

### 챕터 완성의 기준

하나의 주 원칙(챕터)이 완성되었다고 판단하는 기준:

#### 필수 기준 (모두 충족해야 함)

- [ ] 위 5개 요소(선언, 근거, 하위 원칙, 세부 항목, 현실 연결)가 모두 포함되어 있다.
- [ ] 번호 체계가 본 Book Bible의 규칙을 따른다.
- [ ] 톤 & 스타일 가이드의 규칙을 준수한다 (1인칭, 직설적, 명령형).
- [ ] "하지 말 것 리스트"의 항목을 위반하지 않는다.
- [ ] 관통 원칙(원칙 4)과의 연결이 명시적이거나 암묵적으로 존재한다.
- [ ] 다른 원칙과의 상호 참조가 자연스럽게 포함되어 있다 (최소 1개).
- [ ] 분량이 Part별 목표 범위 안에 있다.

#### 품질 기준 (심사 시 확인)

- [ ] 선언이 명확하고 기억에 남는가?
- [ ] 근거가 논리적이고 설득력 있는가?
- [ ] 하위 원칙이 선언을 충분히 구체화하는가? (빠진 측면이 없는가?)
- [ ] 세부 항목이 실행 가능한 수준인가? (너무 추상적이지 않은가?)
- [ ] 저자의 목소리가 일관되게 유지되는가?
- [ ] 불필요한 반복이 없는가?
- [ ] 독자가 이 챕터만 읽어도 하나의 완결된 원칙을 이해할 수 있는가?

### 전체 원고 완성 기준

| 기준 | 설명 |
|------|------|
| **구조 완결성** | 모든 파트(I~IV + 성찰 + 부록)가 존재하고, 파트 간 연결고리가 자연스럽다. |
| **톤 일관성** | Part I부터 성찰까지 저자의 목소리가 동일하다. |
| **주제 완결성** | 13개 원칙이 빠짐없이 다루어지고, 상호 참조 구조가 작동한다. |
| **관통 원칙** | 원칙 4가 책 전체를 관통하며, Part IV에서 원칙 간 위계가 명시된다. |
| **분량** | 전체 15,000~20,000 단어. Part II가 가장 두껍고, Part I이 가장 얇다. |
| **부록** | 원칙 요약표가 정확하고, 본문과 일치한다. |

### 분량 가이드

| 파트 | 목표 분량 | 비중 |

| Part II | 6,000~8,000 단어 | ~40% |
