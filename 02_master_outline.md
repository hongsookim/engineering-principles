#### Chapter: 왜 원칙인가

- **Goal**: 독자가 "원칙 없는 판단은 체계적으로 틀린다"는 것을 납득하고, 이 책을 읽을 동기를 갖는다.
- **Word count target**: 1,200 단어
- **구성**: 3개 섹션 (자기소개가 아니라 원칙의 기원 이야기)

- **Argument beats**:

  1. **실행자에서 판단자로** (~400 단어)
     - 논점: 코드를 짜는 사람에서 판단을 내리는 사람으로의 전환은 역할의 변화가 아니라 사고 체계의 변화다
     - 핵심 서사: Head of Data → CDAO 전환 과정에서의 깨달음. 기술적으로 최선인 판단이 항상 최선은 아니었다는 경험
     - 현실 연결: 토스에서의 Data First → AI Native 전환 배경 (배경만 간결하게 설정)
     - 지적 긴장: "좋은 판단의 공통점은 명확한 원칙에 기반해 있었다. 나쁜 판단의 공통점은 감각에 의존해 있었다."

  2. **원칙이 필요한 순간** (~400 단어)
     - 논점: 기술 리더의 하루는 판단의 연속이며, 수백 개의 판단이 축적되어 조직의 방향이 된다
     - 전개: 하루 동안의 판단 시나리오 나열 — 기술 도입, 장애 대응, 채용 판단, 속도 vs 품질
     - 핵심 선언: "원칙은 판단을 빠르고, 일관되게 만드는 **운영 체제**다"

  3. **왜 원칙인가** (~400 단어)
     - 논점: 기술은 변하지만 판단의 본질은 변하지 않으며, AI 시대에 원칙은 더 중요해졌다
     - 달리오 인용 + 재정의: "원칙은 기술이 변해도 변하지 않는, 판단의 기준이다"
     - 핵심 선언: "불완전한 원칙이라도, 원칙이 없는 것보다 언제나 낫다"
     - 겸손의 선언: 이 원칙들이 완벽하지 않음을 인정. 현실에서 진화해야 함을 명시

- **Connection to next**: "경험에서 원칙이 나왔다. 그 원칙을 공유한다." → Part II의 첫 번째 원칙(현실 직시)으로 자연스럽게 연결. "모든 판단의 출발점은 현실을 있는 그대로 보는 것이다."

### Part II: 엔지니어링 원칙

> **파트 역할**: 개인 차원의 판단 기준 제시. 기술 리더 한 사람이 따르는 원칙.
> **독자 여정**: "판단을 내릴 때 이 프레임워크를 적용할 수 있다"
> **파트 도입문**: 짧은 선언 (~100 단어). "엔지니어링 원칙은 특정 기술이나 시대에 종속되지 않는다."
> **핵심 아크**: 현실 인식(1) → 문제 해결 프로세스(2) → 본질 구별(3) → 책임의 원칙(4) → 판단의 체계화(5)

#### Chapter: 원칙 1 — 현실을 직시하고 대응하라

- **Goal**: 독자가 "모든 판단의 출발점은 현실을 왜곡 없이 보는 것"임을 납득한다. 감이 아니라 데이터로, 보고 싶은 것이 아니라 있는 그대로.
- **Word count target**: 1,500 단어
- **Sub-principles**: 4개 하위 원칙

| 하위 원칙 | 핵심 논점 | 세부 항목 수 | 분량 |
|-----------|----------|------------|------|
| 1.1 데이터를 통해 현실을 보라 | 감은 체계적으로 틀린다. 주요 의사결정은 데이터 근거를 요구하라 | 3 (a~c) | ~350 단어 |
| 1.2 기술 부채를 숨기지 마라 | 부채 은폐는 이자 누적. 의식적 부채와 무의식적 부채를 구별하라 | 3 (a~c) | ~300 단어 |
| 1.3 나쁜 소식이 빨리 올라오는 구조를 만들어라 | 리더의 반응이 조직의 투명성 수준을 결정한다 | 3 (a~c) | ~300 단어 |
| 1.4 자신의 판단도 현실 점검의 대상이다 | "내가 틀렸다면 어떤 증거가 나를 설득할 수 있는가?" | 2 (a~b) | ~250 단어 |

- **5대 필수 요소 체크**:
  - 선언: "현실을 왜곡하는 것이 최악의 판단이다"
  - 근거: 왜곡된 현실 위의 판단은 반드시 더 큰 고통으로 돌아온다
  - 하위 원칙: 4개 (데이터, 부채, 나쁜 소식, 자기 점검)
  - 세부 항목: 각 하위 원칙에 2~3개
  - 현실 연결: 기술적으로 완벽하지만 가치를 만들지 못한 프로젝트 경험

- **Argument beats**:
  1. 도입 선언 — 현실 왜곡이 모든 판단 오류의 시작
  2. 1.1 — 감 vs 데이터. "내 경험상"의 위험성. 단, 데이터 만능주의 경계
  3. 1.2 — 기술 부채 인정의 중요성. 전략적 부채 vs 방만한 부채
  4. 1.3 — 나쁜 소식의 구조적 전달. 리더의 반응이 조직 문화를 결정
  5. 1.4 — 가장 어려운 현실 직시: 자기 자신. 반대 의견을 의식적으로 찾는 습관

- **원칙 간 참조**: 원칙 6(투명성), 원칙 7(실수 학습), 원칙 5(판단 기록)와 상호 강화
- **관통 원칙(4) 연결**: 현실을 왜곡하면 이해 없이 판단하게 된다 — 원칙 4의 전제 조건
- **Connection to next**: "현실을 직시했다면, 다음은 그 현실의 문제를 체계적으로 푸는 것이다." → 원칙 2(5단계 프로세스)로 연결

#### Chapter: 원칙 2 — 다섯 단계 프로세스를 통해 문제를 해결하라

- **Goal**: 독자가 문제 해결의 5단계 순환(정의→인식→진단→설계→실행)을 체화하고, 단계를 건너뛰는 것이 왜 같은 실수를 반복하게 하는지 이해한다.
- **Word count target**: 1,700 단어
- **Sub-principles**: 5개 하위 원칙

| 하위 원칙 | 핵심 논점 | 세부 항목 수 | 분량 |
|-----------|----------|------------|------|
| 2.1 문제를 먼저 정의하라 | 해결책이 아니라 문제가 먼저. "망치를 들면 모든 것이 못으로 보인다" | 4 (a~d) | ~350 단어 |
| 2.2 문제를 회피하지 말고, 인식하라 | 문제를 덮는 것은 문제를 키우는 것. 인식과 즉시 해결은 다르다 | 3 (a~c) | ~280 단어 |
| 2.3 근본 원인을 진단하라 | 표면적 증상이 아니라 구조적 원인을 파고든다. "왜?"를 다섯 번 물어라 | 4 (a~d) | ~320 단어 |
| 2.4 해결책을 설계하라 | 가장 흥미로운 해결책이 아니라 가장 효과적인 해결책. 오버 엔지니어링 경계 | 3 (a~c) | ~280 단어 |
| 2.5 끝까지 실행하라 | 실행되지 않은 설계는 가치가 없다. 완료의 기준을 미리 정의하라 | 4 (a~d) | ~320 단어 |

- **5대 필수 요소 체크**:
  - 선언: "5단계를 따르는 리더는 일관되게 좋은 판단을 내리고, 건너뛰는 리더는 같은 실수를 반복한다"
  - 근거: 달리오의 5단계 순환 + 엔지니어링 맥락으로의 재구성
  - 하위 원칙: 5개 (정의→인식→진단→설계→실행)
  - 세부 항목: 각 하위 원칙에 3~4개
  - 현실 연결: 해결책부터 선택하는 기술 리더의 전형적 실수 사례

- **Argument beats**:
  1. 도입 — 달리오의 5단계 순환 소개 + 핵심은 "순환"한다는 것
  2. 2.1 — 해결책 먼저의 유혹. "이 기술로 무엇을 하지?" vs "어떤 문제를 풀어야 하지?"
  3. 2.2 — 문제 회피의 패턴들: "나중에", "아직 괜찮다", "시간이 해결해줄 것"
  4. 2.3 — 5 Whys. 증상 vs 원인. 응급 처치와 정밀 검사의 구분
  5. 2.4 — 문제 크기에 맞는 해결책. 트레이드오프의 인식
  6. 2.5 — 아이디어의 묘지. 기술 데모 vs 상용화. 중단의 용기

- **원칙 간 참조**: 원칙 5(판단 체계화), 원칙 11(조직 개선)과 상호 강화
- **관통 원칙(4) 연결**: 2.5의 "끝까지 실행"은 원칙 4의 "이해한 것만 배포"와 긴장 — 속도 vs 이해의 균형
- **Connection to next**: "문제를 체계적으로 풀었다면, 다음은 무엇에 집중할지의 기준이다. 변하는 것과 변하지 않는 것을 구별해야 한다." → 원칙 3으로 연결

#### Chapter: 원칙 3 — 변하는 것과 변하지 않는 것을 구별하라

- **Goal**: 독자가 기술 변화 앞에서 유행 추종과 변화 거부의 양극단을 피하고, "원칙이라는 필터"로 기술 선택을 판단하는 프레임을 갖춘다.
- **Word count target**: 1,400 단어
- **Sub-principles**: 4개 하위 원칙

| 하위 원칙 | 핵심 논점 | 세부 항목 수 | 분량 |
|-----------|----------|------------|------|
| 3.1 기술에 매료되지 말고, 문제에 집중하라 | "얼마나 흥미로운가"가 아니라 "얼마나 효과적인가" | 3 (a~c) | ~300 단어 |
| 3.2 새로운 기술이 등장할 때마다, 원칙으로 돌아오라 | 기술 패닉의 패턴은 반복된다. 원칙이라는 필터를 통과시켜라 | 4 (a~d) | ~350 단어 |
| 3.3 변하지 않는 것에 투자하라 | 베조스: "10년 후에도 변하지 않을 것에 투자하라". 사람에 대한 투자 | 3 (a~c) | ~300 단어 |
| 3.4 기술의 변화 속에서 원칙의 일관성을 유지하라 | 원칙이 기술을 흡수해야지, 기술에 종속되면 안 된다 | 2 (a~b) | ~250 단어 |

- **5대 필수 요소 체크**:
  - 선언: "변하는 것과 변하지 않는 것을 구별하지 못하면, 유행 추종이나 변화 거부에 빠진다"
  - 근거: 기술 패러다임이 반복적으로 바뀌어도 좋은 엔지니어링의 본질은 유지됨
  - 하위 원칙: 4개 (문제 집중, 원칙 필터, 불변 투자, 일관성)
  - 세부 항목: 각 하위 원칙에 2~4개
  - 현실 연결: AI 등장 시 조직의 패닉 경험, Data First → AI Native 전환 사례

- **Argument beats**:
  1. 도입 — 기술 변화와 본질의 구별이 기술 리더의 핵심 능력
  2. 3.1 — 엔지니어의 자연스러운 성향(기술 매료)에 대한 경계

#### Chapter: 원칙은 어떻게 작동하는가

- **Goal**: 독자가 원칙을 경직된 교리가 아니라 "살아 있는 알고리즘"으로 이해하고, 원칙 간 충돌 시 판단 기준(원칙 4 최상위)을 체화하며, 원칙을 진화시키는 메타 능력을 갖춘다.
- **Word count target**: 2,500 단어
- **구성**: 3개 섹션

- **섹션 1: 원칙은 알고리즘이다** (~700 단어)
  - 논점: 달리오의 "원칙 = 반복 상황에 대한 알고리즘" 비유. 기술 리더에게 특히 유용
  - 전개: if-then-else로 판단을 체계화. 하지만 예외 처리도 필요. 맹목적 준수 = 원칙 부재와 동일한 위험
  - 핵심 선언: "원칙을 맹목적으로 따르는 것은 원칙을 가지지 않는 것만큼이나 위험하다"

- **섹션 2: 원칙 간의 충돌을 다루는 법** (~800 단어)
  - 논점: 원칙의 위계를 이해하는 것이 핵심. 원칙 4가 최상위
  - 전개:
    - 충돌의 예시: 2.5(끝까지 실행) vs 2.5(중단의 용기). "속도" vs "이해"
    - 원칙 4의 최상위성: 책임에 대한 원칙이므로 다른 원칙보다 우선
    - 충돌은 실패가 아니라 더 깊은 판단이 필요하다는 신호
  - 세부 항목: 3개 (a~c)
  - 현실 연결: 실제 원칙 충돌 상황과 해결 경험

- **섹션 3: 원칙을 진화시키는 법** (~800 단어)
  - 논점: 원칙은 완성형이 아니다. 현실이 원칙을 시험하고, 시험의 결과가 원칙을 갱신한다
  - 전개:
    - 원칙이 현실과 맞지 않을 때: 원칙이 잘못된가, 적용이 잘못된가?
    - 유행에 따른 변경 vs 경험에서 배운 진화의 구분
    - 원칙의 버전 관리. 정기적 재검토 (연 1회)
  - 세부 항목: 4개 (a~d)
  - 현실 연결: 원칙을 수정한 실제 경험과 그 과정의 기록

- **Argument beats (전체 챕터)**:
  1. 도입 — "원칙을 아는 것과 적용하는 것 사이의 간극"
  2. 알고리즘 비유 — 판단의 자동화. 하지만 경직이 아닌 유연함
  3. 충돌의 인식 — 원칙들은 때로 서로 충돌한다. 이것은 자연스럽다
  4. 위계의 이해 — 원칙 4의 최상위성. 왜 책임이 최우선인가
  5. 진화의 방법론 — 원칙은 살아 있는 체계. 삭제, 추가, 갱신
  6. 마무리 — "원칙은 완성형이 아니다. 현실에서 진화한다. 그것이 원칙의 힘이다."

- **관통 원칙(4) 최종 정리**: 이 챕터에서 원칙 4의 위계적 최상위성을 명시적으로 선언하고 근거를 제시

## 원칙 간 연결 구조 — 왜 이 순서인가

### Part II 내부: 원칙 1~5의 순서 논리

```
원칙 1 (현실 직시) ──▶ 원칙 2 (체계적 해결) ──▶ 원칙 3 (본질 구별)
    │                      │                      │
    │  "현실을 보았으니      │  "문제를 풀었으니      │  "집중 대상을 알았으니
    │   문제를 풀어야 한다"   │   무엇에 집중할지      │   책임을 져야 한다"
    │                      │   결정해야 한다"       │
    ▼                      ▼                      ▼
                                              원칙 4 (관통 원칙: 책임)
                                                  │
                                                  │  "책임의 기준 위에서
                                                  │   판단을 반복 가능하게"
                                                  ▼
                                              원칙 5 (판단 체계화)
```

- **원칙 1이 원칙 2보다 먼저**: 현실을 왜곡하면 문제 정의 자체가 틀어진다. 인식이 해결보다 선행.
- **원칙 2가 원칙 3보다 먼저**: 체계적으로 문제를 풀 줄 알아야, "무엇에 집중할지" 구별할 수 있다.
- **원칙 3이 원칙 4보다 먼저**: 변하는 것과 변하지 않는 것을 구별해야, "이해했는가?"를 정확히 판단할 수 있다.
- **원칙 4가 원칙 5보다 먼저**: 책임의 원칙이 세워져야, 판단의 체계화에 의미가 있다. 무책임한 판단의 체계화는 위험하다.

### Part III 내부: 원칙 6~13의 순서 논리 (문화→사람→시스템)

```
문화(토양)                 사람(씨앗)              시스템(구조)
┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐
│ 6. 투명성         │  │ 9. WHO > WHAT    │  │ 11. 조직=기계     │
│ 7. 실수 학습      │──▶│ 10. 훈련/성장     │──▶│ 12. 빌더         │
│ 8. 자율과 책임    │  │                  │  │ 13. 도구/프로토콜  │
└──────────────────┘  └──────────────────┘  └──────────────────┘
```

- **문화가 사람보다 먼저**: 문화(토양)가 갖춰지지 않으면 좋은 사람(씨앗)이 와도 자라지 못한다.
- **사람이 시스템보다 먼저**: 올바른 사람이 없으면 어떤 시스템도 제대로 구축되지 않는다.
- **이 순서가 중요한 이유**: 문화 없이 사람을 세울 수 없고, 사람 없이 시스템을 구축할 수 없다. 역순으로 가면 형식만 남고 실질이 사라진다.

### Part II → Part III 연결

- **논리**: 개인의 원칙이 아무리 훌륭해도, 조직의 구조와 문화가 뒷받침하지 않으면 실현되지 않는다.
- **관통 원칙(4)의 확장**: Part II에서는 개인의 책임, Part III에서는 조직의 책임으로 범위가 확장된다.
